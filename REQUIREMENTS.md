# REQUIREMENTS.md

## 1. Project Overview
The objective is to build a containerized, RAG-based chat application that allows a user to ingest a Wikipedia article, view a concise summary, and interact with the content via a chat interface[cite: 2]. The core focus of this project is not just the functional application, but demonstrating an AI-agent driven engineering workflow encompassing rigorous planning, test-driven development, and modular system design[cite: 2].

## 2. Functional Requirements
* **Data Ingestion:** The application must provide a simple UI form to accept a Wikipedia article URL[cite: 2].
* **Web Scraping:** The backend must scrape the provided URL, extracting the text body, relevant sections, and reasonable references[cite: 2].
* **Summarization:** Upon scraping, the system must generate and display a concise summary of the article using a local LLM[cite: 2].
* **Vector Processing:** The scraped content must be logically chunked, passed through an embedding model, and stored in a vector database[cite: 2].
* **RAG Chat Interface:** The UI must include a chat box below the summary for user queries. Answers generated by the local LLM must be strictly grounded in the retrieved document chunks (RAG pattern) and refrain from utilizing the model's generalized pre-training knowledge[cite: 2].

## 3. Non-Functional Requirements
* **AI-Assisted Workflow:** Planning, decomposition, and execution must be heavily driven by an AI agent (e.g., Cursor), evidenced by clear artifact files (`REQUIREMENTS.md`, `DESIGN.md`, `TASKS.md`)[cite: 2].
* **Local Inference:** The final running application must utilize a local LLM (via Ollama or vLLM) for summarization and chat generation. Hosted APIs (OpenAI, Gemini, etc.) are strictly prohibited in the production container[cite: 2].
* **Test Coverage:** Application code must have a minimum of 85% line coverage[cite: 2]. The LLM and Vector DB must be mocked at the unit test layer, alongside at least one end-to-end integration test validating the wired-up stack[cite: 2].
* **Containerization:** The entire architecture (Web App, Vector DB, and LLM runtime) must spin up using a single command (e.g., `docker compose up`)[cite: 2]. No secrets or API keys may be committed to the repository[cite: 2].

## 4. Out of Scope
To maintain a tight, well-tested implementation, the following features are intentionally excluded[cite: 2]:
* User authentication and account management[cite: 2].
* Session persistence, chat history persistence across reloads, or multi-article databases[cite: 2].
* Complex visual aesthetics or UI animations (focus is on functional, clean UX)[cite: 2].
* Hosted embedding models for production (local embedding models will be prioritized for full offline capability)[cite: 2].

## 5. Assumptions & Unilateral Decisions
* **Tech Stack:** I have selected Node.js for the backend and a minimal React/Next.js setup for the frontend[cite: 2]. This stack offers the fastest iteration speed for web-based LLM applications and provides excellent tooling for test coverage (e.g., Jest/Vitest).
* **LLM Selection:** Given the constraint of running the full stack locally via Docker Compose, I will use `llama3.2:3b` via Ollama[cite: 2]. A 3B parameter model strikes the optimal balance between reasoning capability for the RAG pipeline and reliable performance on standard developer hardware.
* **Vector Database:** I will utilize ChromaDB[cite: 2]. It is highly optimized for localized, containerized environments and requires minimal boilerplate to wire into a Node.js backend compared to heavier alternatives like Milvus.
* **Scraping Strategy:** Given Wikipedia's highly structured and predictable DOM, standard HTTP requests paired with a DOM parser (like Cheerio) will be sufficient. Headless browsers (like Puppeteer) introduce unnecessary container bloat and latency and will be avoided.
