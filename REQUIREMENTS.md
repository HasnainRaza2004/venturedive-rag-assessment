# REQUIREMENTS.md

## 1. Project Overview

The objective is to build a containerized, RAG-based chat application that allows a user to ingest a Wikipedia article, view a concise summary, and interact with the content via a chat interface. The core focus is not only the functional application, but demonstrating an AI-agent driven engineering workflow: rigorous planning, test-driven development, and modular system design.

**Implementation status (reviewed against brief):** All functional requirements below are implemented. Docker Compose E2E verified (ingest, summary, grounded chat). Backend unit coverage ≥85% (~94% lines); integration E2E against Ollama+Chroma completed; coverage screenshot prepared for submission (`docs/coverage/`).

## 2. Functional Requirements

* **Data Ingestion:** The application must provide a simple UI form to accept a Wikipedia article URL.
* **Web Scraping:** The backend must scrape the provided URL, extracting the text body, relevant sections, and reasonable references.
* **Summarization:** Upon scraping, the system must generate and display a concise summary of the article using a local LLM.
* **Vector Processing:** The scraped content must be logically chunked, passed through an embedding model, and stored in a vector database.
* **RAG Chat Interface:** The UI must include a chat box below the summary for user queries. Answers generated by the local LLM must be strictly grounded in the retrieved document chunks (RAG pattern) and refrain from utilizing the model's generalized pre-training knowledge.

## 3. Non-Functional Requirements

* **AI-Assisted Workflow:** Planning, decomposition, and execution must be heavily driven by an AI agent (e.g., Cursor), evidenced by clear artifact files (`REQUIREMENTS.md`, `DESIGN.md`, `TASKS.md`).
* **Local Inference:** The final running application must utilize a local LLM (via Ollama or vLLM) for summarization and chat generation. Hosted APIs (OpenAI, Gemini, etc.) are strictly prohibited in the production container.
* **Test Coverage:** Application code must have a minimum of 85% line coverage. The LLM and Vector DB must be mocked at the unit test layer, alongside at least one end-to-end integration test validating the wired-up stack.
* **Containerization:** The entire architecture (Web App, Vector DB, and LLM runtime) must spin up using a single command (e.g., `docker compose up`). No secrets or API keys may be committed to the repository.

## 4. Out of Scope

To maintain a tight, well-tested implementation, the following features are intentionally excluded:

* User authentication and account management.
* Session persistence, chat history persistence across reloads, or multi-article databases.
* Complex visual aesthetics or UI animations (focus is on functional, clean UX).
* Hosted embedding models for production (local embedding models are used for full offline capability).

## 5. Assumptions & Unilateral Decisions

* **Tech Stack:** Node.js (Express) backend and minimal **React + Vite** frontend (static build served by nginx in Docker). Jest + Supertest for backend tests.
* **LLM Selection:** `llama3.2:3b` via Ollama — balance of RAG quality and local hardware.
* **Embeddings:** `nomic-embed-text` via Ollama.
* **Vector Database:** ChromaDB in Docker with persistent volume.
* **Scraping Strategy:** HTTP + Cheerio on Wikipedia’s structured DOM; no headless browser.
